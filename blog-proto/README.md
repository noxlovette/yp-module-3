# blog-proto

Protobuf-описания и сгенерированные Rust-биндинги для blog-сервиса. Это общий контракт, которым пользуются `blog-server` (реализует его), `blog-client` (обращается к нему по gRPC и HTTP) и `blog-cli` (использует его транзитивно через `blog-client`).

## Что здесь есть

- `proto/blog/v1/blog.proto` — описание `BlogService`: `Register`, `Login`, а также CRUD и `ListPosts` для постов, плюс типы сообщений `Post`/`User`.
- `build.rs` — запускается на этапе компиляции через `tonic-prost-build` и генерирует gRPC-стабы клиента/сервера и структуры сообщений в `OUT_DIR`. Также он навешивает `#[derive(serde::Serialize, serde::Deserialize)]` на все сгенерированные типы и подключает `crate::timestamp_serde` к трём полям `created_at`/`updated_at`.
- `src/lib.rs` — реэкспортирует сгенерированный код и определяет `timestamp_serde` — кастомный (де)сериализатор, который связывает два представления этих полей на проводе:
  - **gRPC / proto**: `int64` unix-таймстамп (в proto3 нет нативного типа времени);
  - **HTTP / JSON**: строка в формате RFC3339 (именно её отдают Actix-хендлеры `blog-server` и ожидает `HttpClient` из `blog-client`).

  Благодаря этому *одна и та же* сгенерированная структура `Post`/`User` корректно сериализуется для обоих транспортов без отдельных DTO.

## Сборка

Кроме рабочего `protoc` (либо встроенного в `tonic-prost-build`, либо установленного в системе) ничего дополнительно не требуется — `cargo build -p blog-proto` перегенерирует биндинги из `proto/blog/v1/blog.proto` при каждом его изменении (см. `cargo:rerun-if-changed` в `build.rs`).

## Как добавить/изменить RPC или сообщение

1. Отредактируйте `proto/blog/v1/blog.proto`.
2. Если новому полю нужно преобразование unix-секунды ↔ RFC3339, добавьте соответствующий вызов `.field_attribute(...)` в `build.rs`.
3. Выполните `cargo build` — сгенерированный модуль (`blog.v1.rs`) не коммитится в репозиторий, так что зависимым крейтам достаточно пересобраться, чтобы подхватить изменения.
